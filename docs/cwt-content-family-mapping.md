# CWTools EU4 schema to `ContentFamilyDescriptor` mapping plan

This document surveys the CWTools EU4 `.cwt` schema set and proposes how foch should absorb the parts that fit its AST + sparse-schema design. The goal is not to import CWTools' dense validator. It is to use CWTools as a high-quality map of EU4 content roots, resource identity rules, and scope-bearing wrapper blocks.

The CWTools EU4 config repository describes itself as `.cwt config files for Europa Universalis IV` and instructs users to place the files in `.cwtools` for VS Code validation ([README.md lines 1-6](https://github.com/cwtools/cwtools-eu4-config/blob/master/README.md#L1-L6)). Its `folders.cwt` file enumerates the major mod roots, including most `common/*` gameplay roots, `decisions`, `events`, `history/*`, `map`, `missions`, `gfx`, `interface`, `localisation`, and DLC metadata ([folders.cwt lines 1-116](https://github.com/cwtools/cwtools-eu4-config/blob/master/folders.cwt#L1-L116)).

## 1. CWTools schema overview

### What `.cwt` files express

CWTools schemas are dense, content-aware rule files. The facts most relevant to foch are:

| CWTools fact | Examples | Productive foch use |
| --- | --- | --- |
| File/root ownership | `type[event]` uses `path = "game/events"` ([events/events.cwt lines 4-8](https://github.com/cwtools/cwtools-eu4-config/blob/master/events/events.cwt#L4-L8)); `type[mission_series]` and `type[mission]` use `path = "game/missions"` ([missions/missions.cwt lines 3-10](https://github.com/cwtools/cwtools-eu4-config/blob/master/missions/missions.cwt#L3-L10)). | `ContentFamilyPathMatcher`, descriptor `id`, and `ModuleNameRule`. |
| Resource identity hints | Events declare `name_field = "id"` ([events/events.cwt lines 4-7](https://github.com/cwtools/cwtools-eu4-config/blob/master/events/events.cwt#L4-L7)); trade nodes declare `name_field = "location"` ([common/trade_consolidated.cwt lines 44-47](https://github.com/cwtools/cwtools-eu4-config/blob/master/common/trade_consolidated.cwt#L44-L47)); government ranks use `name_from_file = yes` and `type_per_file = yes` ([common/governments_and_reforms.cwt lines 52-57](https://github.com/cwtools/cwtools-eu4-config/blob/master/common/governments_and_reforms.cwt#L52-L57)). | `MergeKeySource::FieldValue`, `AssignmentKey`, `ContainerChildKey`, or a future extractor-specific identity rule. |

| Scope metadata | Event subtypes push country/province scopes ([events/events.cwt lines 14-20](https://github.com/cwtools/cwtools-eu4-config/blob/master/events/events.cwt#L14-L20)); many blocks use `replace_scope`, e.g. new diplomatic action blocks are country/country/FROM country ([common/diplomatic_actions_new.cwt lines 46-67](https://github.com/cwtools/cwtools-eu4-config/blob/master/common/diplomatic_actions_new.cwt#L46-L67)); CWTools defines scopes including Country, Province, Trade Node, Unit, Religion, Culture, etc. ([scopes.cwt lines 2-46](https://github.com/cwtools/cwtools-eu4-config/blob/master/scopes.cwt#L2-L46)). | `ContentFamilyScopePolicy` (`root_scope`, `from_alias`, `dynamic_scope`) and extractor wrapper tables for trigger/effect blocks. foch currently only has Country, Province, Unknown, so Trade Node and other scopes remain Unknown unless the core scope enum expands. |
| Trigger/effect/modifier aliases | Event `trigger`, `immediate`, `option`, and `after` blocks use `alias_name[trigger]`/`alias_name[effect]` ([events/events.cwt lines 132-187](https://github.com/cwtools/cwtools-eu4-config/blob/master/events/events.cwt#L132-L187)); scripted triggers/effects are generic aliases over trigger/effect bodies ([common/scripted_triggers_and_effects.cwt lines 14-20](https://github.com/cwtools/cwtools-eu4-config/blob/master/common/scripted_triggers_and_effects.cwt#L14-L20)). | Resource extractor rules and call-edge recognition, not typed validation of every child key. |
| Enumerations and value sets | `enum[power_categories]` is defined as `ADM`, `DIP`, `MIL` ([enums.cwt lines 31-35](https://github.com/cwtools/cwtools-eu4-config/blob/master/enums.cwt#L31-L35)); CB/war-goal files define peace-option enums ([common/casus_belli_and_war_goals.cwt lines 35-83](https://github.com/cwtools/cwtools-eu4-config/blob/master/common/casus_belli_and_war_goals.cwt#L35-L83)). | Generally not descriptor material. Occasionally useful as a reference-target family (`<trade_good>`, `<mission>`, `<wargoal_type>`), but not as a parser constraint. |
| Cardinality, required fields, ranges, severities | Events require one or more `option` blocks ([events/events.cwt lines 141-182](https://github.com/cwtools/cwtools-eu4-config/blob/master/events/events.cwt#L141-L182)); decisions annotate `major`, `potential`, `allow`, `effect`, AI weights, and numeric ranges ([decisions/decisions.cwt lines 18-65](https://github.com/cwtools/cwtools-eu4-config/blob/master/decisions/decisions.cwt#L18-L65)). | Mostly **do not absorb**. They are validation facts, not merge/resource-identity facts. Use only as risk signals for merge/list policies. |
| Localisation expectations | Events map `title` to localisation ([events/events.cwt lines 9-12](https://github.com/cwtools/cwtools-eu4-config/blob/master/events/events.cwt#L9-L12)); missions require title/desc loc keys ([missions/missions.cwt lines 11-19](https://github.com/cwtools/cwtools-eu4-config/blob/master/missions/missions.cwt#L11-L19)). | Do not put in `ContentFamilyDescriptor`; foch tracks localisation separately. |

foch's descriptor boundary is intentionally much smaller. `ContentFamilyDescriptor` stores id/matcher, `ScriptFileKind`, module naming, scope policy, capabilities, extractor, merge key source, and merge policies ([content_family.rs lines 502-515](../crates/foch-language/src/analyzer/content_family.rs#L502-L515)). Capabilities include the coverage flags and dedup-safety opt-ins such as `cross_file_dedup_safe`; the tables below record both `cross_file_dedup_safe` and the requested `per_entry_dedup_safe` recommendation so the design remains valid across the capability-field rollout.

## 2. Coverage gap analysis

`docs/project-status.md` currently lists the semantic-complete gameplay roots in the verified probe ([project-status.md lines 85-147](./project-status.md#L85-L147)) and records the latest baseline as `parse_only = 60`, `semantic_complete = 69` ([project-status.md lines 148-152](./project-status.md#L148-L152)). The table below treats a root as a **gap** when CWTools has schema coverage but the root is not in that semantic-complete list, or when foch only has a generic descriptor without a content-specific extractor/symbol kind. Some rows already have generic `eu4_profile.rs` entries; those are descriptor **promotions**, not brand-new path classifiers.

The raw status-list delta also includes roots that already have explicit foch descriptors or separate handling (`events`, `decisions`, `common/on_actions`, `common/defines`, `customizable_localization`, `gfx`, `interface`, `music`, `sound`, `tutorial`) and roots whose later project-status notes indicate recent promotion (`common/advisortypes`, `common/government_names`, `common/government_ranks`). Those are not prioritized below unless CWTools reveals a descriptor-field change rather than a simple coverage-accounting discrepancy.

Descriptor shorthand:

- `AK` = `MergeKeySource::AssignmentKey`
- `FV(x)` = `MergeKeySource::FieldValue("x")`
- `CCK` = `MergeKeySource::ContainerChildKey`
- `scope(C)` / `scope(P)` / `scope(U)` = Country / Province / Unknown root scope
- `country_from(C)` means root Country with FROM Country; `dynamic` means runtime caller supplies scope
- `xfd` = `cross_file_dedup_safe`; `ped` = `per_entry_dedup_safe`

### Medium-impact simple global-key gaps

These roots are likely safe `AssignmentKey` semantic promotions with small extractors. They already appear in CWTools' folder map but are not in the status semantic-complete list: `common/ai_army`, `common/ai_attitudes`, `common/ai_personalities`, `common/ancestor_personalities`, `common/client_states`, `common/country_colors`, `common/custom_country_colors`, `common/custom_ideas`, `common/dynasty_colors`, `common/estate_crown_land`, `common/estates_preload`, `common/insults`, `common/leader_personalities`, `common/natives`, `common/opinion_modifiers`, `common/region_colors`, `common/ruler_personalities`, `common/units_display` ([folders.cwt lines 3-95](https://github.com/cwtools/cwtools-eu4-config/blob/master/folders.cwt#L3-L95)). Notable evidence:

- `common/opinion_modifiers` has a simple `type[opinion_modifier]` path and scalar body ([common/opinion_modifiers.cwt lines 1-47](https://github.com/cwtools/cwtools-eu4-config/blob/master/common/opinion_modifiers.cwt#L1-L47)).

## 4. Things to NOT absorb

1. **Per-symbol typed schemas.** foch should not validate every allowed key, value type, range, cardinality, or required field. CWTools' event and decision rules are excellent dense validators ([events/events.cwt lines 47-187](https://github.com/cwtools/cwtools-eu4-config/blob/master/events/events.cwt#L47-L187), [decisions/decisions.cwt lines 18-65](https://github.com/cwtools/cwtools-eu4-config/blob/master/decisions/decisions.cwt#L18-L65)); importing that would make foch's parser/diff/patch/merge layers content-sensitive.
2. **Localisation key requirements.** CWTools records expected localisation keys for events, missions, trade goods, and many other roots. foch already has a separate localisation analyzer and merge path; descriptors should not depend on localised key names.
3. **Dense enum membership.** Enums such as `power_categories` and peace options are useful reference material but should not become parse-time constraints ([enums.cwt lines 31-35](https://github.com/cwtools/cwtools-eu4-config/blob/master/enums.cwt#L31-L35), [common/casus_belli_and_war_goals.cwt lines 35-83](https://github.com/cwtools/cwtools-eu4-config/blob/master/common/casus_belli_and_war_goals.cwt#L35-L83)).
4. **Per-content dataclass models.** A Rust struct per EU4 root would violate the AST + sparse-schema invariant. Prefer descriptor + tiny extractor helpers that emit resource definitions and call edges from stable path/identity/scope facts.
5. **Umbrella roots.** Do not add generic `common/` or `map/` descriptors. They obscure coverage accounting and can swallow roots that should be independently classified.
6. **Unverified dedup/noop policies.** Never set `cross_file_dedup_safe`, future `per_entry_dedup_safe`, or any future `per_entry_noop` merely because a root has named entries. Aggregate roots (`missions`, `on_actions`, `tradenodes`, GUI) can make duplicate-looking entries meaningful.

## 5. Implementation roadmap

### Landing order

1. **Callable and modifier resources:** `common/scripted_effects`, `common/triggered_modifiers`, `common/static_modifiers`, `common/timed_modifiers`. These give immediate graph and overlap value with small descriptor/extractor changes.
2. **High-traffic country gameplay:** `missions`, `common/governments`, `common/imperial_incidents`, `common/incidents`, `common/imperial_reforms`, `common/wargoal_types`.
3. **Trade/economic roots:** `common/tradegoods`, `common/tradecompany_investments`, `common/trading_policies`, then `common/tradenodes` last because `name_field = "location"` and route lists make it structurally riskier.
4. **Map/colonial/simple global-key roots:** `common/colonial_regions`, `common/centers_of_trade`, `common/opinion_modifiers`, personality/AI roots, color/client-state roots.
5. **Support/UI/DLC roots:** only after gameplay roots are green or a user-facing conflict report needs them.

### Risks and guardrails

- **Dedup safety is semantic, not syntactic.** Identical-looking entries can be additive in `on_actions`, slot-sensitive in `missions`, route-sensitive in `tradenodes`, or layout-sensitive in UI. Default `xfd=false`, `ped=false` unless the game resolves entries in a global key namespace and file location/order cannot change meaning.
- **CWTools identity hints are not always merge keys.** `name_field = "location"` for trade nodes is a resource identity signal, but merge materialization may still need assignment-key context or block replacement.
- **Scope mismatch.** CWTools models scopes that foch does not (`Trade Node`, `Religion`, `Culture`, etc.; [scopes.cwt lines 2-46](https://github.com/cwtools/cwtools-eu4-config/blob/master/scopes.cwt#L2-L46)). Until foch's `ScopeType` grows, keep these roots `Unknown` and only push Country/Province where already supported.
- **Schema drift and spelling differences.** Use CWTools as a map, not an oracle. Verify against installed EU4 data before each root promotion.
- **Overfitting dense schemas.** The implementation slice should only add root matching, resource identity, sparse wrapper recognition, and merge/dedup policy. Do not add a validation model.
- **Coverage accounting.** Every root promotion should update semantic-index tests, base-data coverage assertions, run the real EU4 probe, and record the new baseline in `docs/project-status.md`.

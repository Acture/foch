# 2026-05 architecture review

Reviewed at `0da9de9` after the alpha-1 fleet landing.

## 1. Crate boundary integrity
- **Current state:** The dependency direction is mostly intact: `foch-engine` depends only on `foch-language` and `foch-core`, `foch-language` depends only on `foch-core`, and I found no `foch-language -> foch-engine` or upward `foch-core` imports (`crates/foch-engine/Cargo.toml:10-33`, `crates/foch-language/Cargo.toml:10-22`, `crates/foch-core/Cargo.toml:10-25`). The boundary break is presentation logic living in engine: `foch-engine` now owns `ratatui`/`crossterm`, exposes `tui_conflict_handler`, and the CLI reaches into `foch_engine::merge::conflict_handler` to set interactive mode (`crates/foch-engine/Cargo.toml:31-32`, `crates/foch-engine/src/merge/mod.rs:1-25`, `crates/foch-cli/src/cli/handler/merge.rs:8-9`).
- **Recommendation:** Move terminal UI and interactive-mode wiring out of `foch-engine` into `foch-cli` (or a dedicated frontend adapter) and leave engine with a frontend-agnostic conflict-resolution interface only. After that split, make `tui_conflict_handler` and interactive config setters `pub(crate)`.
- **Severity:** **ARCH-BUG**

## 2. Module shape inside `foch-engine`
- **Current state:** The top-level flow is coherent: `workspace::resolve` builds the workspace and mod snapshots, `run_checks` consumes `workspace + runtime + merge::{dag,namespace}`, `merge::execute` layers cache/materialization/revalidation, and `graph`/`simplify` are mostly consumers of `workspace` and `runtime` (`crates/foch-engine/src/workspace/resolve.rs:53-140`, `crates/foch-engine/src/run_checks.rs:1-25`, `crates/foch-engine/src/merge/execute.rs:1-17`, `crates/foch-engine/src/graph/export.rs:117-127`, `crates/foch-engine/src/simplify/execute.rs:16-23`). The debt is concentration: `merge/` is ~19.7k LoC, with `patch_merge.rs`, `materialize.rs`, `tui_conflict_handler.rs`, and `dag.rs` all large, and even non-merge code reuses `merge::emit` (`crates/foch-engine/src/runtime/binding.rs:1-3`, `crates/foch-engine/src/simplify/execute.rs:4-7`).
- **Recommendation:** Keep the current product-line split, but pull general Clausewitz emission out of `merge/` and plan a second split inside `merge/` along pipeline seams: planning, patch engine, resolution/policy, and output/materialization. The current file names are coherent, but the subtree is one more growth cycle away from becoming the default dumping ground.
- **Severity:** **ARCH-DEBT**

## 3. Public API surface
- **Current state:** `foch-engine/src/lib.rs` re-exports nearly every subsystem, and `merge/mod.rs` publicly exposes conflict handlers, handler registry, patch machinery, and the TUI module (`crates/foch-engine/src/lib.rs:1-39`, `crates/foch-engine/src/merge/mod.rs:1-25`). The CLI already depends on internal-looking paths such as `foch_engine::merge::conflict_handler`, `foch_engine::cache::*`, `foch_engine::base_data::*`, and `foch_engine::workspace::FileFilter`, so module layout is turning into API contract (`crates/foch-cli/src/cli/handler/merge.rs:8-9`, `crates/foch-cli/src/cli/handler/cache.rs:6-10`, `crates/foch-cli/src/cli/handler/data.rs:7-12`, `crates/foch-cli/src/cli/handler/data.rs:56-58`). The recent split into `MergeStatusView` and `AnalysisStatusView` helps, but `foch_core::model::MergeReport` is still growing into a catch-all payload (`crates/foch-engine/src/merge/execute.rs:46-62`, `crates/foch-core/src/model/merge.rs:188-238`).
- **Recommendation:** Introduce a narrow engine facade for CLI-facing operations and make low-level cache helpers, conflict handlers, and TUI modules non-public. Also treat `MergeReport` as an artifact schema rather than the only status API; if beta adds more counters, split summary views from deep diagnostics instead of widening the root struct again.
- **Severity:** **ARCH-BUG**

## 4. `ContentFamily` as the extension point
- **Current state:** `ContentFamilyDescriptor` is still the right extension seam, and the two new dedup flags are conceptually coherent: one gates cross-file namespace dedup and one gates same-file vanilla-equivalent entry pruning (`crates/foch-language/src/analyzer/content_family.rs:255-268`, `crates/foch-engine/src/merge/materialize.rs:483-493`, `crates/foch-engine/src/merge/materialize.rs:1201-1211`). The problem is representational: the behavior bag is now a set of booleans plus a dedicated builder convenience, so the next few “safe only when load semantics prove X” flags will sprawl (`crates/foch-language/src/analyzer/content_family.rs:510-589`, `crates/foch-language/src/analyzer/eu4_profile.rs:15-35`, `crates/foch-language/src/analyzer/eu4_profile.rs:144-205`).
- **Recommendation:** Before adding more capability flags, group dedup/load-semantics into a nested policy object or feature enum. `BlockPatchPolicy` still looks like the right model for now: `Recurse` as default plus key-specific `Union` overrides is still understandable and matches the current country-history use (`crates/foch-language/src/analyzer/content_family.rs:204-235`, `crates/foch-language/src/analyzer/eu4_profile.rs:79-84`, `crates/foch-language/src/analyzer/eu4_profile.rs:276-276`).
- **Severity:** **ARCH-DEBT**

## 5. Cache pipeline coherence
- **Current state:** The new engine cache stack is internally understandable: mod snapshots use rkyv under `mods/`, patch/dag-base layers use bincode under `diffs/` and `dag-base/`, and modset results use tar.gz + JSON under `modsets/` (`crates/foch-engine/src/cache/mod.rs:1-27`, `crates/foch-engine/src/cache/mod_parse_cache.rs:22-37`, `crates/foch-engine/src/cache/mod_diff_cache.rs:1-6`, `crates/foch-engine/src/cache/dag_base_cache.rs:1-6`, `crates/foch-engine/src/cache/modset_cache.rs:1-7`, `crates/foch-engine/src/cache/mod_parse_cache.rs:1043-1055`). The architectural problem is that `foch cache` still straddles two cache systems: stats/list/clean target the engine layers, but `gc` is wired to the older `foch-language` parser cache rooted at `~/.cache/foch/parse_cache/v7`, with its own cap and eviction logic (`crates/foch-cli/src/cli/handler/cache.rs:41-49`, `crates/foch-cli/src/cli/handler/cache.rs:166-185`, `crates/foch-language/src/analyzer/semantic_index/parse_cache.rs:94-111`, `crates/foch-language/src/analyzer/semantic_index/parse_cache.rs:183-257`).
- **Recommendation:** Unify cache ownership before adding more layers: either fold the legacy parser cache into the engine cache tree as an explicit fifth layer, or expose it as such in the CLI until it is retired. Once the control plane is unified, add one size-cap/eviction story for all layers instead of age-only cleanup for engine caches plus byte-cap GC for only the legacy parser cache.
- **Severity:** **ARCH-BUG**

## 6. Conflict resolution chain
- **Current state:** The current chain is explicit and works for v1: `LookupHandler -> DepImpliesResolutionHandler -> DeferHandler`, then a post-pass `prompt_survivors_and_persist` for interactive picks (`crates/foch-engine/src/merge/materialize.rs:1639-1665`, `crates/foch-engine/src/merge/conflict_handler.rs:175-353`, `crates/foch-engine/src/merge/conflict_handler.rs:659-687`, `crates/foch-engine/src/merge/conflict_handler.rs:767-853`). What is missing is a cleaner extension shape: named handlers are still a hard-coded string dispatch table, and `ConflictDecision::PickModWithRecord` fuses audit bookkeeping into the action enum (`crates/foch-engine/src/merge/handler_registry.rs:11-15`, `crates/foch-engine/src/merge/handler_registry.rs:28-51`, `crates/foch-engine/src/merge/conflict_handler.rs:80-97`).
- **Recommendation:** If beta needs additional nontrivial handlers (policy packs, batch-mode resolvers, research handlers), change handlers to return `{decision, audit_record}` and build chains from a list/registry of handlers instead of nested generic pairs. That removes the `PickMod` vs `PickModWithRecord` split without weakening auditability.
- **Severity:** **ARCH-DEBT**

## 7. Test architecture
- **Current state:** The three tiers are mostly stable: source-adjacent unit tests are heavy, fixture-backed integration tests cover merge behavior, and real installed-mod probes already have an `#[ignore]` slot (`crates/foch-engine/src/base_data/mod.rs:2232-2233`, `crates/foch-engine/src/merge/materialize.rs:2216-2216`, `crates/foch-engine/tests/merge_e2e.rs:32-54`, `crates/foch-engine/tests/patch_real_mods.rs:1-8`). The weak point is reproducibility: the N=37 acceptance probe still lives outside cargo’s documented test surface, so an important architecture gate is sitting in session state rather than the repo.
- **Recommendation:** Promote the real-playset probe suite into an ignored cargo test target (preferred) or a bench harness only if you need timing-oriented output. That keeps the repo’s existing three-tier shape while making the acceptance gate rerunnable by any maintainer.
- **Severity:** **ARCH-DEBT**

## 8. Documentation coherence
- **Current state:** Documentation is split across flat `docs/`, root release-prep files, and until this review no `docs/reviews/` home; `docs/README.md` still points mainly at the older core set (`docs/README.md:7-18`, `KNOWN_ISSUES.md:1-52`, `ALPHA_ANNOUNCEMENT.md:1-60`, `docs/RELEASE_CHECKLIST.md:1-16`). `docs/auto-merge-roadmap.md` is stale after today’s landing: it still lists “Architecture Cleanup R1” as active even though the workspace/runtime/merge/graph/simplify split and `check --graph-out` removal it calls for have already landed (`docs/auto-merge-roadmap.md:65-78`, `docs/auto-merge-roadmap.md:291-297`).
- **Recommendation:** No code change needed now, but beta prep should give release docs and review docs stable homes and rewrite `auto-merge-roadmap.md` from a post-alpha hardening baseline instead of a pre-cleanup one. The main risk here is contributors reading outdated “next milestone” guidance and coding against the wrong architectural phase.
- **Severity:** **ARCH-DEBT**

## Summary
- **ARCH-BUG:** 3
- **ARCH-DEBT:** 5
- **Most urgent fixes before beta debt compounds:** engine/frontend split for conflict UI, cache control-plane unification, and public API contraction around `foch-engine`.
